## Context

当前单模型监控系统存在以下局限：
- 数据库仅记录单一模型的测试结果，无法区分不同模型
- 适配器只选择第一个可用的模型进行监控
- 看板只能展示一个模型的统计数据

需要扩展为多模型监控系统，支持：
- 获取 NewAPI 所有可用模型列表
- 对每个模型独立进行可用性测试
- 在看板上按模型分组展示统计数据

## Goals / Non-Goals

**Goals:**
- 支持同时监控多个模型
- 每个模型独立记录测试数据
- 看板展示模型列表及各模型的独立可用性统计
- 保持 8:00-18:00 工作时间调度逻辑

**Non-Goals:**
- 不实现模型间的负载均衡或自动切换
- 不实现告警通知功能
- 不改变现有的 3 分钟间隔调度逻辑

## Decisions

### 1. 数据库模型变更：增加 model 字段

在 test_records 表增加 model 字段存储测试的模型名称。

**为什么：**
- 需要区分不同模型的测试记录
- 向后兼容：现有数据的 model 字段为 NULL
- 查询时可以通过 model 字段过滤

**替代方案考虑：**
- 新建 model_records 表：通过外键关联，增加复杂度
- 使用 model 作为记录前缀：破坏现有结构，不必要

### 2. 调度策略：串行测试所有模型

调度周期内，依次对每个模型发起可用性测试。

**为什么：**
- 实现简单，易于调试
- 减少对 NewAPI 的并发压力
- 每个模型的测试时间独立，便于统计分析

**替代方案考虑：**
- 并行测试所有模型：增加系统复杂度，可能触发限流
- 按权重分配测试间隔：过度设计，当前场景不需要

### 3. 模型列表获取：启动时获取，按需刷新

在 Agent 启动时获取模型列表，运行期间定时刷新。

**为什么：**
- 模型列表相对稳定，不需要频繁刷新
- 减少 API 调用次数
- 检测到模型不可用时重新获取列表

**替代方案考虑：**
- 每次测试前重新获取：增加延迟，不必要
- 持久化模型列表：增加复杂度，模型可能动态变化

### 4. 看板展示：模型卡片 + Tab 切换

使用模型卡片展示所有模型状态，支持点击查看详情。

**为什么：**
- 一目了然看到所有模型状态
- 操作直观，无需复杂交互
- 便于快速定位问题模型

## Risks / Trade-offs

[Risk] 模型数量较多时测试时间长 → Mitigation：设置单模型超时（5秒），总调度超时不影响下次调度

[Risk] 数据库查询性能随数据量增长下降 → Mitigation：已有30天数据清理策略，model 字段建立索引

[Risk] 模型不可用时重试导致调度延迟 → Mitigation：单模型测试失败直接记录，不重试到其他模型

## Migration Plan

1. 数据库迁移：ALTER TABLE 添加 model 列（向后兼容）
2. 修改 adapter.test_connection() 支持指定模型
3. 修改 agent 对每个模型发起测试
4. 修改 database 查询方法支持 model 过滤
5. 修改 web API 返回分组数据
6. 更新前端看板展示
7. 测试验证

## Open Questions

2. 历史数据处理：NULL model 的旧数据如何展示？（建议归类为"未知"）
