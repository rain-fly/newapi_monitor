## Context

当前需要建立一个自动化的 NewAPI 模型可用性监控系统。该系统需要每3分钟对 NewAPI 接口发起一次最小化的可用性测试，记录测试结果，并在看板上展示不同时间窗口的可用性统计数据。测试任务仅在每天 8:00-18:00 执行。

技术约束：
- 使用 Python 作为主要开发语言
- 数据存储采用 SQLite，轻量化且无需额外服务
- 看板使用 Web 页面展示，便于嵌入或独立部署

## Goals / Non-Goals

**Goals:**
- 每3分钟自动发起一次 API 可用性测试
- 仅在 8:00-18:00 时间段内执行测试
- 记录每次测试的时间戳、可用状态、延迟、错误信息
- 计算并展示 3分钟/30分钟/3小时/6小时/12小时/24小时 各时间窗口的可用性
- 提供直观的 Web 看板展示实时和历史数据

**Non-Goals:**
- 不实现告警通知功能（可后续扩展）
- 不对接除 NewAPI 以外的其他 API 提供商
- 不支持多模型并行测试（当前仅针对单一模型）

## Decisions

### 1. 整体架构：Agent + Web Server 混合模式

采用定时触发的独立 Agent 进程执行测试 + 轻量 Web 服务展示看板的架构。

**为什么：**
- 测试任务需要定时触发，使用独立 Agent 可以与 Web 服务解耦
- Web 服务专注于数据展示，不需要持续运行
- 便于后续扩展为分布式部署

**替代方案考虑：**
- 单一服务同时处理调度和展示：增加复杂度，单点故障风险更高
- 纯 cron + 静态页面：数据动态性差，无法满足实时展示需求

### 2. 数据存储：SQLite

使用 SQLite 作为本地数据库存储测试记录。

**为什么：**
- 零配置，部署简单
- 适合中等规模数据量（每小时20条，每天200条）
- 无需额外服务进程

**替代方案考虑：**
- JSON 文件：简单但不便于查询和统计
- PostgreSQL/MySQL：过度设计，增加部署复杂度

### 3. 看板技术：Flask + ECharts

使用 Flask 提供轻量 Web 服务，ECharts 实现数据可视化。

**为什么：**
- Flask 轻量，学习成本低
- ECharts 图表丰富，支持实时更新
- 便于嵌入到现有系统

**替代方案考虑：**
- React/Vue：过度复杂，本项目不需要状态管理
- 纯静态页面：无法实现数据动态加载

### 4. 调度方式：schedule 库 + 守护进程

使用 Python schedule 库实现定时调度，运行在守护进程模式下。

**为什么：**
- 纯 Python 实现，无需额外依赖（如 cron）
- 代码内聚，便于理解和维护
- 支持灵活的时间规则配置

**替代方案考虑：**
- 系统 cron：依赖操作系统，跨平台性差
- APScheduler：功能更强大但更复杂，本项目不需要

### 5. API 测试方式：最小 Token 请求

使用尽可能小的输入输出 token 进行测试。

**为什么：**
- 减少 API 调用成本
- 降低测试本身对系统的影响
- 更快速地完成可用性检测

## Risks / Trade-offs

[Risk] 网络波动导致偶发失败 → Mitigation：增加重试机制（最多3次），但记录重试次数以便区分真实故障和网络问题

[Risk] SQLite 在高并发写入时性能下降 → Mitigation：本项目写入频率极低（每3分钟1次），不存在并发写入问题

[Risk] Web 服务和 Agent 进程需要同时运行 → Mitigation：提供统一的启动脚本，通过进程管理确保两者协同工作

[Risk] 数据长期积累导致数据库膨胀 → Mitigation：设置数据保留策略，自动清理超过30天的历史数据
